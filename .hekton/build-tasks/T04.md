## Task Description

Implement typed-placeholder + HMAC keying logic, most likely as a new module
`crates/vg-core/src/keying.rs` (vg-core is where Task T02 put all other shared/canonical
logic -- see `crates/vg-core/src/{types,traits,api}.rs` -- and Task T05's not-yet-built
`vg-vault` crate will later call into this from its `VaultStore::intern` implementation;
this task does not need `vg-vault` to exist to be buildable or testable standalone).
`interface-contracts.md`'s `VaultStore` doc comment is the authority on the formula: "stable
placeholder via salted HMAC over `(canonical(value), ty, ns)`" -- see
`crates/vg-core/src/traits.rs` around the `VaultStore` trait and `crates/vg-core/src/types.rs`
for `Placeholder { display: String, mapping_ref: MappingRef }`, `EntityType`, and `Namespace`.

Concretely, this task needs:

1. A `canonicalize(value: &str) -> String` (or similar) normalisation step before hashing --
   e.g. trimming and case-folding where the entity type calls for it -- so trivial
   formatting differences (whitespace, case) don't produce different placeholders for what
   is semantically the same value.
2. A salted HMAC keying function over `(canonical(value), EntityType, Namespace)` (the
   `hmac` + `sha2` crates are reasonable choices; add as `vg-core` dependencies if not
   already present) producing a stable, deterministic key: the same input always yields the
   same key, and a value interned under one `Namespace` must key differently than the same
   value under a different one (this task does not implement namespace *isolation on lookup*
   -- that's `VaultStore::resolve`'s job, in T05 -- but the keying function itself must be
   namespace-sensitive as an input, or T05 has nothing to isolate).
3. Per-type ordinal assignment so a `Placeholder`'s `display` string looks like `EMAIL_001`,
   `ACCOUNT_ID_014` (see `README.md`'s own example) -- the first time a distinct key is seen
   for a given `EntityType` within a `Namespace`, it gets the next sequential number for that
   type; the same key seen again returns the same ordinal.
4. Luhn checksum (for card-number-shaped values) and mod-97 checksum (for IBAN-shaped values)
   validators, used so a placeholder's own display value can be checked (or constructed) to
   remain checksum-valid -- a masked pack that replaces a card number with something that
   fails Luhn on its face is a worse debugging experience than one that doesn't.
5. A session-scoped in-memory cache (a `HashMap` keyed on the same inputs as step 2 is
   reasonable) so repeated keying of the same value within one run doesn't recompute the HMAC
   each time. This is a fast front-cache, not a replacement for T05's persistent vault.

This is a one-shot headless dispatch: there is no follow-up channel. If any exact naming,
module boundary, or type-signature choice above is ambiguous, use your best judgment, follow
the existing conventions already frozen in `crates/vg-core` and `interface-contracts.md`, and
record the assumption in the PR description and `docs/decisions.md` rather than asking a
clarifying question -- an unasked-but-documented judgment call is recoverable in review; a
session that ends by asking a question is not.
