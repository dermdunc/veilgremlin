## Task Description

Implement `vg_core::traits::VaultStore` in `crates/vg-vault/src/` (currently an empty stub
crate). The trait: `fn intern(&self, value: &Secret, ty: EntityType, ns: &Namespace) ->
Result<Placeholder, VaultError>`, `fn resolve(&self, p: &Placeholder, ns: &Namespace) ->
Result<Secret, VaultError>`, `fn purge_expired(&self) -> Result<usize, VaultError>` (see
`crates/vg-core/src/traits.rs`).

Concretely:
1. SQLCipher-backed storage -- `rusqlite` with a bundled-sqlcipher feature is a reasonable
   choice (add as a dependency); schema needs at least a `mapping` table (encrypted
   value <-> placeholder, namespace, entity type, created/expiry timestamps) and a
   `demask_event` table.
2. OS-keychain key wrap, macOS first -- the `keyring` crate (or `security-framework`
   directly) is a reasonable choice for wrapping the DB encryption key; never persist it
   plaintext.
3. `Secret` (from `vg-core`) already zeroizes on drop -- use it for anything holding a raw
   value; don't reintroduce a separate non-zeroizing string path.
4. `intern` **must call into `crates/vg-core/src/keying.rs`'s `Keyer`/`placeholder_key`**
   (Task T04, already merged) for the actual keying formula, rather than reimplementing
   HMAC/canonicalisation logic here. **Hard requirement found during T04's own review:**
   whatever `Keyer` instance backs this vault must have its per-`(Namespace, EntityType)`
   ordinal counters reseeded from this vault's own persisted `mapping` rows at
   construction time -- a fresh in-memory `Keyer` after a process restart must not hand
   out a display ordinal (e.g. `EMAIL_001`) that collides with or diverges from what the
   vault already recorded for a different value. Do not skip this.
5. Namespace isolation is part of the contract, not a convention: `resolve` must return
   `VaultError::NotFound` (not distinguish "wrong namespace" from "doesn't exist") when
   `ns` doesn't match the namespace a placeholder was interned under. Use
   `vg_core::conformance::assert_vault_roundtrip` -- it checks this explicitly.
6. TTL/purge: `purge_expired` removes rows past their TTL. Prepared-statement reuse: cache
   prepared statements on the struct rather than re-preparing per call (this is on the
   hot lookup path per the interface contract).

This is a one-shot headless dispatch: there is no follow-up channel. If any exact
naming/schema/crate choice above is ambiguous, use your best judgment, follow existing
`vg-core`/`interface-contracts.md` conventions, and record the assumption in
`docs/decisions.md` rather than asking a clarifying question.
