## Task Description

Implement `vg_core::traits::AuditSink` in `crates/vg-audit/src/` (currently an empty stub
crate). The trait: `fn write(&self, event: AuditEvent) -> Result<AuditId, AuditError>`,
`fn get(&self, id: AuditId) -> Option<AuditEvent>` (see `crates/vg-core/src/traits.rs`
and `audit.rs` for the `AuditEvent` enum: `Scan`, `PolicyDecision`, `MappingCreated`,
`Block`, `DemaskRequest`, `DemaskDecision`, plus others -- check the actual enum
definition, the interface-contracts.md sketch is illustrative not exhaustive).

Append-only storage: a simple append-only file (JSON Lines, one event per line, fsynced)
is a reasonable, dependency-light choice; SQLite (matching T05's vault) is also
reasonable if you want one storage technology across both. Either is fine -- pick one and
record the choice, don't ask.

**The hard, security-relevant contract:** no raw value in any `AuditEvent` variant, ever
-- only refs/counts/versions. Test this with
`vg_core::conformance::assert_audit_event_excludes_raw_values`, which checks both the
literal raw value AND its `Debug`-escaped form (a raw value containing control
characters renders escaped in `{event:?}`, so checking only the unescaped literal would
miss exactly that class of leak -- read the helper's own doc comment before assuming a
simpler check suffices). Version every record type so the schema can evolve later without
breaking old records.

This is a one-shot headless dispatch: there is no follow-up channel. If any exact
naming/storage/schema choice above is ambiguous, use your best judgment, follow existing
`vg-core`/`interface-contracts.md` conventions, and record the assumption in
`docs/decisions.md` rather than asking a clarifying question.
