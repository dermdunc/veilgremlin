## Task Description

Implement the `vg` CLI (`crates/vg-cli`) and the Claude Code hook adapter
(`crates/vg-adapters-claude`), wiring the now-complete `vg_core::{scan, mask, rehydrate}`
pipeline (contract v1.1). The adapter contract is `interface-contracts.md` section 8; follow it
exactly (hooks: `UserPromptSubmit` -> mask(prompt), `PreToolUse`/`PostToolUse` -> mask(tool_io);
exit codes 0 pass-through / 2 transformed / 1 block-with-reason-to-stderr; the adapter NEVER
calls `vault.resolve` -- demask is only the separate user-invoked `vg demask` flow).

**A known contract gap you will hit immediately -- resolve it via the contract-change protocol
(v1.1 -> v1.2, `agent-factory-plan.md` section 6, documented in interface-contracts.md, never
silently):** the frozen `rehydrate(masked: &str, dest, actor)` cannot reach the vault, and the
T07 review banked a HARD requirement: demask must resolve **exclusively via the pack's
`MappingRef`s -- never by pattern-scanning text for placeholder-shaped strings** (input already
containing `EMAIL_001`-shaped text is indistinguishable from pipeline output). Since
`MaskedPack.mapping_refs` carries refs but not the display<->ref association needed to locate
placeholders, the recommended v1.2 shape is: (a) `MaskedPack` gains an additive field binding
each display string to its `MappingRef` (populated by `mask` -- it has both at intern time),
and (b) `rehydrate` is re-signed to take the pack + `&Policy` + `&Namespace` + dest + actor,
keeping the existing hard-deny check FIRST (before any vault/policy consultation), then
`policy.engine.demask_allowed`, then per-ref `policy.vault.resolve`, substituting only the
displays the pack itself minted. Document the inherent residual: raw input that collides with
a pack-minted display substitutes too -- that is in-band text masking's nature, recorded not
hidden. Use your judgment on exact shapes; record every call in docs/decisions.md.

CLI subcommands (clap is already in the workspace lock via criterion; add it as a real
dependency): `vg run -- <cmd...>` (wrapper: writes a hook-config for the wrapped Claude Code
invocation, prints the pre-send summary from MaskStats, passes through env for the Bedrock
path -- configuring the wrapped CLI, e.g. CLAUDE_CODE_USE_BEDROCK=1 pass-through, is the
"Bedrock masked-request path"; do NOT build an HTTP client); `vg inspect <file>` (what WOULD
be masked -- runs scan+policy classification, prints findings/classes, changes nothing);
`vg diff --masked <file>` (show original vs masked); `vg demask --from <file> --to <dest>`
(the explicit gate above); `vg audit last|<id>` (read the audit sink); `vg policy check`
(load + validate the layered packs, print the resolved summary); `vg vault stats`
(mapping_count, no values). Every command help text complete.

State/wiring: the hooks and CLI need a consistent place for the vault DB, audit log, policy
layers, and persisted packs (demask needs the pack from a prior mask). A repo-local
`.veilgremlin/` state dir (gitignored) with the global policy at a well-known path is a
reasonable choice; hooks read JSON on stdin per Claude Code's hook protocol and must emit
ONLY the transformed text on stdout when exiting 2. Keep per-hook-invocation overhead
minimal and measured -- hooks sit on the interactive hot path, and a cold SQLCipher open +
keychain fetch per invocation is exactly the kind of perceptible friction the acceptance
criterion below exists to catch; record cold-vs-warm numbers where feasible.

Tests: integration tests driving the compiled `vg` binary (assert_cmd or std::process) over a
temp state dir with the real crates -- a masked round trip (mask via hook-shaped input, then
`vg demask --to local_patch` restores the original); the demask gate DENIES
remote_model_prompt/observability_sink regardless of actor; a spoofed placeholder-shaped
string NOT minted by the pack is left untouched by demask; `vg inspect` output contains
classes and counts but never a raw detected value where the class is redact/mask.

This is a one-shot headless dispatch: no follow-up channel. Use your judgment on exact
naming/shapes, follow the frozen contracts and existing conventions, record every assumption
in docs/decisions.md rather than asking. NOTE: the human-in-the-loop UX session in the
acceptance criteria happens at review time with the human -- your job is to make it possible
(working hooks + a docs/runbook-hooks.md walkthrough for wiring them into a real Claude Code
session), not to claim it happened.
