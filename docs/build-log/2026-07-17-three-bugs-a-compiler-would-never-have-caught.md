# Three bugs a compiler would never have caught

**2026-07-17**

The keying module from the previous entry compiled clean and passed every test on the first real run. That's a good result, and also not the whole story. Tests only catch what someone thought to write a test for, and a second, colder pair of eyes — a Codex review, run specifically because it shares none of the assumptions whoever wrote the code was working under — found three real bugs sitting quietly inside code that was, by every test it had, correct.

The first was a collision. The keying formula folds an entity's type into the hash alongside the value, so the same value under two different types produces two different placeholders. Except for one type: a policy-dictionary class called `Custom`, identified by a free-text name. The code that turned that name into a hash input first upper-cased it and collapsed punctuation for display purposes — a sensible thing to do for a human-readable label. The bug was reusing that same collapsed string as the cryptographic input. `"foo-bar"`, `"foo_bar"`, and `"foo bar"` are three different policy classes that a person could reasonably define separately, and all three collapsed to the identical tag. Three different things, one hash. Fixed by giving the display formatting and the cryptographic input their own separate functions, which also closes a second problem for free: a future cosmetic rename of a display prefix can no longer silently change what every existing placeholder's hash actually was.

The second was about what "the same value" means in practice. An IBAN written as `GB29NWBK60161331926819` and the same IBAN written as `GB29 NWBK 6016 1331 9268 19` are the same bank account to any human reading them, and the detector built back in T03 already treats them as the same finding. The keying code did not agree with either of them: it collapsed whitespace but never stripped it, so the compact and the spaced form of one real value hashed to two different placeholders. Same bug, same shape, for sort codes and phone numbers. This is exactly the kind of thing that looks fine in isolation — every individual function did what its own docstring said — and only shows up once you ask the actual question the acceptance criterion asks: does the same real value get the same placeholder. It didn't, for anything with optional formatting. Fixed by stripping the formatting noise for the three types where it's genuinely cosmetic, and left alone for types like IP addresses and postcodes where the separators are structural, not decorative.

The third was smaller: a debug-printing implementation that, with the best of intentions, printed the entire cryptographic key in full whenever anything tried to inspect it — a test failure message, a stray log line. Not a leak of anyone's real data, but exactly the kind of internal material this whole project exists to keep off of a shared surface. Fixed to redact, the same way the project's own `Secret` type already did.

None of these three would have been caught by asking "does it compile" or even "do the tests pass," because the tests that existed were testing the things the code's author already thought to test. That's not a criticism of the author — it's the entire argument for a second, differently-built reviewer looking at security-relevant code before it ships, the same discipline this project has leaned on since the very first detector.

One more thing came out of fact-checking the dispatch's own self-reported summary against what was actually true by the end of the day: that summary said it had recorded five judgment calls in the decisions log. It had recorded six. A small, honest miscount, not a real problem — but it's a good reminder that even a careful self-report is a claim, not a fact, and the discipline that catches a real cryptographic bug is the same discipline that catches a wrong number in a sentence.
